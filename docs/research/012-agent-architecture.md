# Agent Architecture

Date: 2026-06-19

## Question

What should `betternat-agent` do at runtime, and how should it interact with the Terraform provider, cloud SDKs, nftables, eBPF, and observability systems?

## Short Answer

Current decision as of 2026-06-25:

`betternat-agent` is now responsible for a LoxiLB-first datapath. It must
reconcile LoxiLB egress SNAT rules and re-export LoxiLB-derived metrics.
nftables is not a product fallback; older nftables-first examples below are
design history unless explicitly updated.

Superseded fallback note: BetterNAT no longer has a product fallback datapath.
The current sources of truth are `docs/architecture.md`, `docs/spec-v0.md`,
and `docs/research/055-no-nftables-fallback-decision.md`. Older fallback
language in this document is design history only.

`betternat-agent` is the runtime control plane on each NAT appliance.

It should own:

- local datapath readiness,
- HA state machine,
- lease renewal,
- cloud failover operations through official SDKs,
- route/EIP verification,
- LoxiLB health and rule reconciliation,
- nftables/sysctl fallback health,
- optional eBPF flow accounting,
- Prometheus metrics,
- local doctor API.

It should not own:

- initial infrastructure provisioning,
- long-term Terraform state,
- broad discovery of every VPC route table,
- UI/dashboard rendering,
- package installation after boot except controlled upgrades.

Terraform provider creates and configures the system. Agent keeps it healthy and handles runtime failover.

## Responsibility Split

```text
Terraform provider:
  declarative product resource
  cloud resource create/update/delete/import
  initial route and public IP configuration
  IAM/RBAC and lease backend creation
  passes signed/rendered agent config

betternat-agent:
  local NAT appliance runtime
  HA ownership and lease renewal
  route/EIP failover via SDK
  local datapath checks
  metrics and flow accounting
  runtime status

betternat CLI:
  discovery helper
  doctor/debug UX
  local admin commands
  rollback helper
  does not replace Terraform provider
```

## Process Model

Run one systemd service:

```text
betternat-agent.service
```

Subcomponents inside the process:

```text
config loader
cloud provider client
lease manager
HA state machine
datapath manager
public identity manager
health checker
metrics exporter
flow accounting manager
event logger
local admin API
```

Keep one process for v0. Splitting into multiple daemons increases operational surface too early.

## HA State Machine

States:

```text
INIT
STANDBY
TAKING_OVER
ACTIVE
DEMOTING
ERROR
MAINTENANCE
```

### INIT

Actions:

- load config,
- identify instance/ENI through metadata service,
- initialize cloud SDK client,
- verify IAM/RBAC,
- verify source/destination check disabled if cloud exposes it,
- render/load nftables rules,
- verify sysctls,
- initialize metrics.

Exit:

- to `ACTIVE` if current lease and route target belong to this node,
- to `STANDBY` otherwise.

### STANDBY

Actions:

- monitor lease owner,
- monitor active health,
- keep local datapath ready,
- serve metrics,
- do not mutate routes unless takeover conditions pass.

Takeover trigger:

- active health failed,
- lease expired or acquire condition passes,
- local datapath ready,
- cloud API reachable,
- configured quorum/guard checks pass.

### TAKING_OVER

Actions:

1. Acquire lease with new generation.
2. Ensure local datapath ready.
3. If shared EIP mode, associate EIP to self.
4. Replace private route target to self.
5. Verify route table state.
6. Verify EIP/public identity state.
7. Run outbound probe.
8. Confirm lease generation still current.
9. Transition to `ACTIVE`.

Ordering between EIP and route should be configurable/tested. For stable egress IP, EIP-before-route may avoid new flows leaving through the wrong public IP.

### ACTIVE

Actions:

- renew lease,
- export active metrics,
- verify route still points to self,
- verify EIP still belongs to self if configured,
- report datapath health,
- respond to planned demotion.

If lease renewal fails:

- stop active mutations,
- optionally continue forwarding if route still points here and fail-closed would be worse,
- enter `ERROR` or `DEMOTING` depending on policy.

### DEMOTING

Actions:

- stop renewing lease,
- stop claiming active status,
- optionally stop NAT datapath only if traffic should not flow through stale node,
- wait for route/EIP to move away,
- transition to `STANDBY`.

### ERROR

Actions:

- expose metrics,
- log actionable error,
- avoid split-brain,
- keep datapath behavior based on policy:
  - fail-open: continue forwarding if already active route target,
  - fail-closed: stop forwarding to avoid wrong egress identity.

Default should probably be fail-open for datapath but fail-closed for control-plane mutations.

## Cloud Provider Interface

Agent should use official cloud SDKs through an internal interface:

```go
type CloudProvider interface {
    Identity(ctx context.Context) (NodeIdentity, error)
    CurrentRoute(ctx context.Context, ref RouteRef) (RouteState, error)
    ReplaceRoute(ctx context.Context, ref RouteRef, target RouteTarget) error
    CurrentPublicIdentity(ctx context.Context, ref PublicIdentityRef) (PublicIdentityState, error)
    AssociatePublicIdentity(ctx context.Context, ref PublicIdentityRef, target NodeIdentity) error
    VerifyPermissions(ctx context.Context) error
}
```

AWS implementation:

- EC2 metadata / IMDSv2 for identity.
- EC2 SDK for route/EIP/ENI operations.
- DynamoDB SDK for lease backend.

Do not shell out to `aws`.

## Lease Manager

Interface:

```go
type LeaseManager interface {
    Acquire(ctx context.Context, key string, candidate Candidate, ttl time.Duration) (Lease, error)
    Renew(ctx context.Context, lease Lease, ttl time.Duration) error
    Release(ctx context.Context, lease Lease) error
    Current(ctx context.Context, key string) (Lease, error)
}
```

Lease data:

```text
ha_group_id
owner_node_id
generation
expires_at
last_renewed_at
route_targets
public_identity
agent_version
```

Rules:

- all active mutations require current lease generation,
- renew uses conditional update on owner/generation,
- stale active demotes when generation changes,
- TTL is cleanup only, not timing source.

## Datapath Manager

v0 responsibilities:

- enable IP forwarding,
- load nftables rules,
- verify nftables rules,
- verify `nf_conntrack`,
- read conntrack count/max,
- expose datapath health,
- avoid unnecessary rule reloads.

It should not hand-edit arbitrary user firewall rules.

Prefer a dedicated nftables table:

```text
table inet betternat
```

Do not own the whole system firewall unless appliance AMI is explicitly dedicated.

## eBPF Flow Accounting Manager

v1 responsibilities:

- load eBPF objects,
- attach TC programs to configured interfaces,
- pin maps,
- read counters/top-N,
- export bounded metrics,
- detach cleanly on shutdown/upgrade.

Failure policy:

- if eBPF accounting fails, NAT forwarding continues,
- emit degraded observability metric,
- do not fail the appliance.

## Metrics Exporter

Expose Prometheus metrics:

Agent/HA:

- `betternat_agent_up`
- `betternat_active{ha_group}`
- `betternat_lease_generation`
- `betternat_lease_renew_errors_total`
- `betternat_failover_events_total{reason,result}`
- `betternat_cloud_api_latency_seconds{operation}`
- `betternat_route_target_match`
- `betternat_public_identity_match`

Datapath:

- `betternat_bytes_total{direction}`
- `betternat_packets_total{direction}`
- `betternat_drops_total{reason}`
- `betternat_conntrack_entries`
- `betternat_conntrack_limit`
- `betternat_conntrack_insert_failed_total`
- `betternat_interface_drops_total{interface,direction}`

Flow attribution:

- Keep Prometheus cardinality bounded.
- Export top-N summaries or aggregate by subnet/team/workload.
- Do not label every destination IP by default.

## Local Admin API

Expose only on localhost or a protected Unix socket by default.

Useful endpoints/commands:

```text
/healthz
/readyz
/status
/metrics
/debug/routes
/debug/lease
/debug/datapath
```

CLI can call local API:

```sh
betternat status
betternat doctor
betternat failover status
betternat maintenance enter
betternat maintenance leave
```

## Logging and Events

Use structured logs.

Every cloud mutation should log:

- operation,
- resource ID,
- request ID if SDK exposes it,
- lease generation,
- old target,
- new target,
- latency,
- result.

Events should also be exported as metrics and optionally CloudWatch logs.

## Upgrade Behavior

Agent upgrade must be safe:

- active node should renew lease until just before restart,
- standby should not take over for a short planned restart unless health threshold is exceeded,
- rolling upgrade should support planned demotion,
- eBPF program/map version changes must be compatible or cleanly detached.

Need command:

```sh
betternat maintenance enter
```

Possible planned failover:

```sh
betternat failover --planned --to standby
```

## Security

Agent hardening:

- run as root only if required for nftables/eBPF; otherwise split privileges later,
- systemd hardening options,
- IMDSv2 only,
- no inbound admin port by default,
- least-privilege IAM,
- config file mode `0600`,
- avoid logging secrets,
- cloud API calls scoped to tagged resources.

Potential later split:

- privileged local datapath helper,
- unprivileged control-plane/metrics process.

Do not overcomplicate v0 unless the threat model requires it.

## Failure Policies

Define explicitly:

### Control-plane failure

If cloud APIs are unavailable:

- active keeps forwarding,
- active cannot safely renew/takeover forever,
- standby must not take over without lease,
- emit degraded metric.

### Lease backend failure

If active cannot renew lease:

- after grace period, stop active mutations,
- maybe keep forwarding while route points to self,
- do not allow standby takeover unless it can acquire lease.

### Datapath failure

If local NAT datapath fails:

- active should demote if standby is healthy,
- standby can take over after lease rules pass.

### Observability failure

If eBPF/metrics fail:

- do not fail NAT,
- mark observability degraded.

## MVP Agent Scope

v0:

- AWS provider.
- DynamoDB lease.
- route failover.
- optional shared EIP reassociation.
- nftables datapath checks.
- Prometheus metrics.
- local `doctor`.
- structured logs.

v1:

- eBPF flow accounting.
- Kubernetes metadata enrichment client.
- planned maintenance/failover.
- richer cost attribution metrics.

v2:

- dynamic ENI/IP mode.
- multi-cloud providers.
- active-flow preservation experiments.
- split privileged datapath helper if needed.

## Decision

Build `betternat-agent` as a small runtime controller with strict boundaries:

```text
local datapath health
+ HA state machine
+ cloud SDK failover operations
+ lease/fencing
+ metrics/flow accounting
```

Keep infrastructure creation in the Terraform provider. Keep user-facing diagnosis in CLI. Keep all runtime failover independent of Terraform.
