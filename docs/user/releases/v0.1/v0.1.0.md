# BetterNAT v0.1.0 Release Notes

Date: 2026-06-24

## Status

`v0.1.0` is the first BetterNAT GA line for self-managed AWS egress gateway
deployments.

Use:

```hcl
terraform {
  required_providers {
    betternat = {
      source  = "nowakeai/betternat"
      version = "= 0.1.0"
    }
  }
}

resource "betternat_gateway" "egress" {
  # ...
  betternat_version = "v0.1.0"
}
```

## What Changed Since The Alpha Line

- Terraform Registry install is the primary install path.
- BetterNAT uses a cloud-init bootstrap path on a user-selected Linux AMI by
  default; no public BetterNAT AMI is required for 0.1.0.
- Gateway nodes run `betternat-agent`, LoxiLB, the `betternat` CLI, and
  Prometheus metrics.
- Runtime state is coordinated through a DynamoDB-backed coordination table for
  leases, agent registry records, peer discovery, and handover records.
- `betternat status` reads daemon and peer data and prints a human-readable
  fleet view with active/standby role, health, versions, IPs, and basic traffic
  counters.
- Active/standby HA supports route replacement, shared-EIP reassociation when
  `stable_egress_ip=true`, graceful systemd stop lease release, ASG lifecycle
  handling, Spot interruption handling, and proactive handover.
- `associate_public_ip_address` is exposed for advanced launch-template control.
  The provider derives the default from bootstrap mode and stable-EIP mode.
- LoxiLB is pinned by BetterNAT release policy. The initial 0.1.0 line targets
  LoxiLB `v0.9.8.6`.

## Validation

0.1.0 is based on the alpha validation sequence recorded in:

- `docs/research/042-provider-alpha7-clean-aws-validation.md`
- `docs/research/046-provider-alpha8-ga-soak-results.md`
- `docs/research/047-runtime-alpha8-asg-lifecycle-validation.md`

That validation covered Terraform Registry install, cloud-init bootstrap,
active/standby status, stable-EIP egress, non-stable egress, manual handover,
systemd stop release behavior, ASG lifecycle handling, Spot interruption code
path conformance, metrics, destroy, and residual-resource scans.

## Known Limitations

- AWS only.
- One AZ per HA group in 0.1.0. Multi-AZ gateway groups are planned for a later
  release.
- No AWS NAT Gateway equivalent SLA.
- New connections recover after failover; active connections may reset.
- No public BetterNAT AMI is published for 0.1.0.
- The default `cloud_init` path depends on package repositories, Docker image
  pull, GitHub release asset reachability, checksum verification, and cloud-init
  execution.
- Stable EIP mode converges back to the shared EIP, but per-node public IPv4 can
  still exist for bootstrap and management in `cloud_init` mode.
- Strict "every successful sample always returns only the shared EIP" semantics
  are future hardening and likely require a secondary private IP or ENI based
  egress identity.
- High-volume savings are modeled, not proven by expensive multi-TB benchmark
  runs.

Read [Limitations](../../reference/LIMITATIONS.md) before using BetterNAT with
real route tables.

## Upgrade Notes

Changing provider version alone does not upgrade running gateway node binaries.
Use [Upgrade And Replacement Guide](../../operations/UPGRADE_REPLACEMENT_GUIDE.md)
for in-place capacity updates, explicit replacement, and blue/green replacement.

Provider patch releases must not introduce breaking changes. Breaking changes
must be reserved for a later minor or major release and called out in release
notes.

## Artifact Integrity

Release assets should be verified against published SHA256 checksums. BetterNAT
0.1.0 does not require artifact signing beyond checksum verification.
